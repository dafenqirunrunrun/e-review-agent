# Public Runtime Audit

Phase: `v1.9.0 Public Runtime Verification - Phase 2`

## Repository Structure

- Java aggregate backend: `litemall-all`
- Admin API module: `litemall-admin-api`
- Customer API module: `litemall-wx-api`
- Database schema: `litemall-db/sql/litemall_table.sql`
- AI schema and risk tables: `litemall-db/sql/litemall_review_ai_analysis.sql`, `litemall-db/sql/litemall_ai_review_risk_task.sql`
- AI service: `ai-service`
- Admin frontend: `litemall-admin`
- Customer H5 frontend: `litemall-vue`

## Reused Capabilities

- `litemall-all` runs both `/admin/**` and `/wx/**` APIs.
- `/wx/order/comment` creates real `litemall_comment` records.
- `/admin/ai/patrol/run-once` scans `litemall_comment` and persists `litemall_review_ai_analysis`.
- `AiRiskTaskService` creates `litemall_ai_review_risk_task` from persisted analysis.
- `/admin/ai/operation/handle` handles a risk task and records operation logs.
- `/admin/ai/dashboard/summary` reports analysis, patrol, risk and operation status.

## Public AI Runtime Boundary

Public Docker verification uses a deterministic public rule engine so the business workflow can be reproduced without private model assets. This mode does not represent the private local-model or Enterprise RAG runtime.

Expected public mode indicators:

- `runtimeMode = public-rule`
- `engineType = rule`
- `modelLoaded = false`
- `privateAssetsRequired = false`

## Seed Data Boundary

`litemall_public_ci_seed.sql` prepares only prerequisite data:

- public CI admin account
- public CI customer account
- one public seed product
- reviewable order and order-goods rows

It does not insert AI analysis rows, risk tasks or operation logs. Those are generated by the application workflow during smoke tests.

## Missing Capabilities Before Phase 2

- No full public compose stack existed.
- Backend database host was not container-friendly.
- AI service did not expose public runtime liveness/readiness.
- Public runtime E2E was not automated in GitHub Actions.

## Phase 2 Adjustments

- Use `litemall-all` as the backend service.
- Use Nginx containers for admin and customer frontends.
- Use MySQL official image with public schema and seed initialization.
- Use deterministic `public-rule` AI mode.
- Use authenticated real APIs for comment submission, Agent patrol, risk task handling and dashboard checks.
